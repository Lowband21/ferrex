# Agent Orchestration for Parallel Ferrex Development

> Spec for managing concurrent agent-driven implementation across multiple
> worktrees in the Ferrex bare repo.

## Status

| Field | Value |
|---|---|
| Created | 2025-07-15 |
| Phase | Specification |

---

## Problem

Ferrex has 4+ parallel workstreams (foundation, iOS, Android, 10-foot mode,
plus ongoing desktop player work). Each runs in its own git worktree. The goal
is to have concurrent pi agents working on each, with:

1. **Autonomy** — agents know what to work on without babysitting
2. **Shared decisions** — implementation discoveries in one worktree propagate
   to others
3. **Trajectory changes** — a human can redirect any agent's focus without
   disrupting others
4. **Observability** — at any time, a human can see what each agent is doing
   and what it's blocked on

The existing pi extension model (TypeScript, `ExtensionAPI`, tools + prompts +
system prompt injection) provides the mechanism. The question is what
specifically to build.

---

## Decisions: Locked

### D-AO-001: One worktree per workstream, one agent per worktree
- Each parallel effort gets its own git worktree under `/home/lowband/dev/ferrex/`
- Each agent session is launched in its worktree's directory
- Agents never cross worktree boundaries for writes (reads are ok for reference)
- **Rationale:** Git worktrees provide natural isolation. No merge conflicts
  during active development. Each agent's working directory scopes its context.

### D-AO-002: Specs are the shared source of truth
- `docs/specs/*.md` are committed and shared across all worktrees
- Changes to specs happen on a single branch and are cherry-picked/merged
  to worktrees that need them
- Specs define WHAT. Plans (gitignored, local) define HOW.
- **Rationale:** Already established in the mobile-apps specs work.

### D-AO-003: Plans are local, generated by agents, reviewed by human
- Each worktree has `docs/plans/` (gitignored) containing the agent's current
  implementation plan
- Plans are generated by the agent based on specs + current codebase state
- Humans review and approve plans before agents execute
- Plans can be regenerated at any time if trajectory changes
- **Rationale:** Plans are ephemeral strategy documents. Committing them would
  create merge noise and stale artifacts.

## Architecture

### The Bare Repo as Orchestration Hub

```
/home/lowband/dev/ferrex/
├── .bare/                          # Shared git object store
├── main/                           # Main branch (desktop player, server)
│   └── (pi agent: desktop work)
├── mobile-apps/                    # feat/mobile-apps branch
│   └── (pi agent: foundation + specs)
├── ios/                            # feat/ios-app branch (future)
│   └── (pi agent: iOS implementation)
├── android/                        # feat/android-app branch (future)
│   └── (pi agent: Android implementation)
├── 10ft/                           # feat/10ft-mode branch (future)
│   └── (pi agent: 10-foot mode)
└── .ferrex-agents/                 # Agent coordination layer (NEW)
    ├── state/                      # Per-agent state files
    │   ├── mobile-apps.json
    │   ├── ios.json
    │   ├── android.json
    │   └── 10ft.json
    ├── decisions/                  # Cross-cutting decision log
    │   └── decisions.jsonl         # Append-only decision journal
    └── messages/                   # Inter-agent message passing
        ├── inbox-ios.jsonl
        ├── inbox-android.jsonl
        └── inbox-10ft.jsonl
```

### Coordination Layer: `.ferrex-agents/`

Lives outside any worktree (sibling to `.bare/`). Provides:

1. **Agent state files** — each agent writes its current status:
   ```json
   {
     "worktree": "ios",
     "branch": "feat/ios-app",
     "currentPhase": "Phase 3: Library Browsing",
     "currentStep": "3.4: Performance validation",
     "status": "working",
     "blockedOn": null,
     "lastActivity": "2025-07-15T14:30:00Z",
     "recentDecisions": [
       "LazyVGrid sufficient for 60fps - no UICollectionView fallback needed",
       "Nuke memory cache set to 100MB based on profiling"
     ],
     "questionsForHuman": []
   }
   ```

2. **Decision journal** — append-only log of implementation decisions that
   other agents should know about:
   ```jsonl
   {"ts":"2025-07-15T14:30:00Z","agent":"ios","decision":"FlatBuffer UUID fields use [ubyte:16] with helper extensions","affects":["android","foundation"],"spec_ref":"mobile-apps-wire-format.md OQ-001"}
   {"ts":"2025-07-15T15:00:00Z","agent":"android","decision":"OkHttp interceptor pattern for auth works cleanly - recommend similar pattern for iOS URLSession","affects":["ios"],"spec_ref":null}
   ```

3. **Message inbox** — targeted messages between agents or from human:
   ```jsonl
   {"ts":"2025-07-15T16:00:00Z","from":"human","priority":"high","message":"Pause iOS Phase 5 - server streaming endpoint has a bug. Fix landing in main shortly."}
   {"ts":"2025-07-15T16:30:00Z","from":"android","priority":"info","message":"ExoPlayer requires Accept-Ranges header. Verified server sends it correctly for mkv/mp4. iOS agent: AVPlayer should work too."}
   ```

## Pi Extension: `ferrex-agent`

A project-specific pi extension that gives each agent awareness of:
- Its role and current plan
- The shared specs
- What other agents are doing
- Pending messages and decisions

### Extension Tools

#### `ferrex_status`
Show current agent state + all sibling agent states.
```
Usage: ferrex_status
Returns: table of all active worktrees, their current phase, status, and recent decisions
```

#### `ferrex_decide`
Log an implementation decision to the shared decision journal.
```
Usage: ferrex_decide decision="FlatBuffer UUID as [ubyte:16]" affects=["android","ios"]
Returns: confirmation + decision ID
```

#### `ferrex_message`
Send a message to another agent's inbox (or broadcast).
```
Usage: ferrex_message to="android" priority="info" message="AVPlayer confirmed working with range requests"
Returns: confirmation
```

#### `ferrex_inbox`
Check and acknowledge messages in this agent's inbox.
```
Usage: ferrex_inbox [--ack <id>]
Returns: unread messages, ordered by priority then time
```

#### `ferrex_plan`
Read or regenerate the current implementation plan.
```
Usage: ferrex_plan [--regen]
Returns: current plan contents, or triggers plan regeneration from specs
```

#### `ferrex_spec`
Quick access to read specs without navigating the filesystem.
```
Usage: ferrex_spec [name]
Returns: spec contents, or list of available specs
```

#### `ferrex_checkpoint`
Save a progress checkpoint — what's done, what's next, any blockers.
Called periodically by the agent or on human request.
```
Usage: ferrex_checkpoint phase="Phase 3" step="3.4" notes="Grid hitting 55fps, investigating..."
Returns: state file updated
```

### System Prompt Injection

On `before_agent_start`, the extension injects:

```
## Ferrex Agent Context
- Worktree: {worktree_name} ({branch})
- Role: {role_description}
- Current plan: {current_phase} / {current_step}
- Specs: {list of relevant specs}

## Sibling Agents
- ios: {status summary}
- android: {status summary}
- 10ft: {status summary}

## Unread Messages ({count})
{message summaries if any}

## Recent Cross-Cutting Decisions ({count})
{recent decisions from other agents}

## Standing Instructions
1. Work through your plan phase by phase. Don't skip ahead.
2. When you make a decision that affects other workstreams, use ferrex_decide.
3. Check ferrex_inbox periodically for messages from other agents or human.
4. Use ferrex_checkpoint after completing each step to update your status.
5. If blocked, update your state with the blocker and stop — don't guess.
6. Specs are immutable. If you disagree with a spec, flag it as a question
   for human review — don't deviate.
```

### Agent Launch

Each agent is launched with the extension:

```bash
# Terminal 1: Foundation work
cd /home/lowband/dev/ferrex/mobile-apps
pi -e ~/.pi/ferrex-agent/extensions/ferrex-agent.ts \
   --append-system-prompt "You are the Foundation agent. Your role is implementing Plan 00: FlatBuffers schemas, server content negotiation, and build tooling."

# Terminal 2: iOS work (once foundation is ready)
cd /home/lowband/dev/ferrex/ios
pi -e ~/.pi/ferrex-agent/extensions/ferrex-agent.ts \
   --append-system-prompt "You are the iOS agent. Your role is implementing Plan 01: the Ferrex iOS app."

# Terminal 3: Android work (once foundation is ready)
cd /home/lowband/dev/ferrex/android
pi -e ~/.pi/ferrex-agent/extensions/ferrex-agent.ts \
   --append-system-prompt "You are the Android agent. Your role is implementing Plan 02: the Ferrex Android app."

# Terminal 4: 10-foot mode
cd /home/lowband/dev/ferrex/10ft
pi -e ~/.pi/ferrex-agent/extensions/ferrex-agent.ts \
   --append-system-prompt "You are the 10-foot mode agent. Your role is implementing Plan 03: controller-driven UI for ferrex-player."
```

### Human Interaction Patterns

**Redirect an agent:**
```bash
# Send a message to the iOS agent
echo '{"from":"human","priority":"high","message":"Stop Phase 5. Focus on improving grid scroll performance in Phase 3 instead."}' \
  >> /home/lowband/dev/ferrex/.ferrex-agents/messages/inbox-ios.jsonl
```
Or just open the iOS pi session and tell it directly.

**Check on all agents:**
```bash
# Quick status from any worktree
cat /home/lowband/dev/ferrex/.ferrex-agents/state/*.json | jq -r '.worktree + ": " + .status + " - " + .currentStep'
```

**Review decisions:**
```bash
# Last 10 cross-cutting decisions
tail -10 /home/lowband/dev/ferrex/.ferrex-agents/decisions/decisions.jsonl | jq .
```

**Change trajectory:**
1. Edit the relevant spec in one worktree
2. Commit and push to the branch
3. Cherry-pick or merge into affected worktrees
4. Send messages to affected agents about the change
5. Agents regenerate their plans from the updated specs

## Decisions: Open

### O-AO-001: Agent session persistence
- pi supports `--continue` and `--session` for session persistence.
- Should agents maintain long-running sessions, or start fresh each time?
- Long-running: more context, but context window fills up.
- Fresh with checkpoint: less context bloat, but loses conversational state.
- **Recommendation:** Long-running sessions with periodic `--fork` to prune
  context when it gets too large. Checkpoints capture essential state.

### O-AO-002: Automated inter-agent coordination vs. human-mediated
- Should agents automatically act on messages from other agents?
- Or should all cross-agent messages be reviewed by human first?
- **Recommendation:** Start with human-mediated. Agents surface messages but
  don't auto-act on them. Reduce friction as trust builds.

### O-AO-003: Build/test orchestration for non-Rust worktrees
- The existing orca-pilot extension orchestrates Rust cargo builds.
- iOS (Xcode) and Android (Gradle) need different build orchestration.
- Should the ferrex-agent extension include build orchestration per platform,
  or keep it simple and let agents use bash directly?
- **Recommendation:** Start with bash. Add Xcode/Gradle orchestration only if
  the friction becomes a real bottleneck.

### O-AO-004: How to handle shared server changes
- Foundation work (Plan 00) modifies `ferrex-server` and `ferrex-core`.
- These changes need to be available to the desktop player worktree (main).
- Options: merge foundation branch into main, or have the foundation agent
  work directly on main.
- **Recommendation:** Foundation agent works on `feat/mobile-apps` branch.
  Server changes are cherry-picked to `main` when stable. Mobile app worktrees
  branch from `feat/mobile-apps`.

## Implementation Priority

The ferrex-agent extension is built incrementally:

### v0 (minimum viable, build first)
- Agent state files (read/write JSON)
- System prompt injection with worktree context
- `ferrex_status` tool (read all agent states)
- `ferrex_checkpoint` tool (write own state)
- `ferrex_spec` tool (read specs)
- No inter-agent messaging yet — human uses direct pi sessions

### v1 (add coordination)
- Decision journal (`ferrex_decide`)
- Message inboxes (`ferrex_message`, `ferrex_inbox`)
- `ferrex_plan` tool (read/regenerate plans)

### v2 (add automation)
- File watcher for spec changes → alert agent
- Auto-checkpoint on phase transitions
- Dashboard script that shows all agent states in a TUI

---

## Non-Goals

- No centralized "conductor" agent that manages others. Human is the conductor.
- No automatic merging between worktrees. Git operations stay manual/human-initiated.
- No shared runtime state beyond files. Agents coordinate through the filesystem,
  not through shared memory or IPC.
- No attempt to share pi sessions across agents. Each agent is a fully
  independent pi instance.
